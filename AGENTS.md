# AGENTS.md — правила для AI-агентов

Правила для AI-агентов (Codex, OpenCode, Roo, Kilo, Pi и др.) по работе с данным пакетом и его конвенциями.

---

## Проект

**prikotov/coding-standard** — PHP-пакет с двумя частями:

1. **PHPCS Sniffs** — автоматические проверки через PHP CodeSniffer 4.x (`src/`).
2. **Conventions** — документация DDD-конвенций (`docs/conventions/`), копируемая в проект-потребитель через `bin/coding-standard-init`.

---

## Принципы работы с кодом

- **PHP 8.4**, Symfony 8.0, строгая типизация (`declare(strict_types=1)`).
- **Читаемость** важнее производительности. Осмысленные имена, минимальная вложенность, небольшие блоки.
- **Стабильность**: покрывайте ключевую логику тестами, низкая связанность.
- **Умеренность**: не добавляйте технологии и абстракции без необходимости.

Подробнее: [Ценности](docs/conventions/principles/values.md) | [Code Style](docs/conventions/principles/code-style.md).

## Правила написания документации

### Язык и стиль

- **Русский** с английским термином в скобках при первом упоминании.
- Формулировки как «пули» — короткие, чёткие, без воды и дублирования.
- Каждый документ описывает **один подход** или **одну сущность**.
- Ссылки на официальные источники (Symfony docs, Wikipedia, книги по DDD/Clean Code).

### Front matter

Каждый `.md` файл в `docs/conventions/` начинается с YAML front matter:

```yaml
---
name: Command Handler
type: rule
description: Правила создания и использования обработчиков команд
---
```

**Обязательные поля:**

| Поле | Описание |
|------|----------|
| `name` | Название документа на русском или английском |
| `type` | Тип документа: `rule`, `index` или `meta` |
| `description` | Краткое описание содержания документа |

**Типы документов:**

| type | Когда | Проверка секций |
|------|--------|----------------|
| `rule` | Документ-правило (конвенция, паттерн) | Да — обязательные секции |
| `index` | Индексный файл категории (список ссылок) | Нет |
| `meta` | Мета-документ (AGENTS.md, README.md) | Нет |

### Обязательная структура документа-правила (type: rule)

````markdown
---
name: {Название}
type: rule
description: {Описание}
---

# {Заголовок на русском} ({English Term})

**{Определение}** — краткое описание со ссылкой на источник.

## Общие правила

- Чёткие, маркированные пункты: можно/нельзя, именование, внедрение.

## Зависимости *(если применимо)*

- Что можно внедрять. Что запрещено.

## Расположение *(если применимо)*

- Слой (Domain, Application, Infrastructure, Presentation).
- Путь: `ProjectName\Common\Module\{ModuleName}\{Layer}\{Context}\{Name}`

## Как используем

- Правила внедрения и ограничения.

## Пример

```php
<?php
declare(strict_types=1);
// Минимальный рабочий пример — реальный PHP, не псевдокод
```

## Чек-лист для проведения ревью кода

- [ ] {пункт}
- [ ] {пункт}
````

### Форматирование

- **Markdown** для всех документов.
- **Запрещён псевдокод** — только реальный PHP-пример, упрощённый из проекта.
- Комментарии в примерах минимальны, код — самодокументируемый.
- Кодовые блоки всегда с `<?php` и `declare(strict_types=1);`.

## Правила создания/обновления документов

1. **Проверьте существование** — нет ли уже документа на тему в `docs/conventions/`.
2. **Добавьте front matter** — `name`, `type`, `description`.
3. **Выберите категорию**: `layers/`, `core-patterns/`, `modules/`, `ops/`, `testing/`, `principles/`, или создайте новую.
4. **Именование файлов и папок**: kebab-case, lowercase с дефисами (`value-object.md`, `core-patterns/`).
5. **Обновите индекс** — добавьте ссылку в `docs/conventions/index.md`.
6. **Проверьте ссылки** — все относительные ссылки должны быть актуальны.
7. **Примеры кода** — берите из реального проекта, упрощайте.
8. **Запустите валидацию** — `composer validate-docs`.

---

## Инструкция по обновлению конвенций

При обновлении документации в этом пакете:

1. Внесите изменения в `docs/conventions/`.
2. Убедитесь, что ссылки в `index.md` актуальны.
3. Проверьте, что init-скрипт (`bin/coding-standard-init`) корректно копирует новые файлы.
4. Запустите `composer check` для проверки сниффов.
